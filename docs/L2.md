ROS 2’s URDF tutorials are organized exactly in this direction: first build a visual robot model, then make it movable, then clean it up with Xacro, and use `robot_state_publisher` to publish the robot frames for visualization. In ROS 2, launch files are normally kept in a package’s `launch/` directory, and Python launch files should end with `launch.py` so `ros2 launch` recognizes them properly. 

# Lesson 2 — First 1-joint arm in RViz

## Goal of this lesson

By the end, you will have:

* one minimal robot arm
* one revolute joint
* one Xacro file
* one RViz launch file
* one moving joint slider with `joint_state_publisher_gui`

The idea is simple:

```text
base_link --- joint1 --- link1
```

When you move the slider, `joint_state_publisher_gui` publishes joint values, `robot_state_publisher` reads the robot model and those joint states, and then publishes the TF transforms that RViz displays. ([docs.ros.org][2])

# 2. Create the Xacro robot file

Open this file:

```bash
cd ~/mybot_ws/src/my_arm_description/urdf
gedit 1dof_arm.urdf.xacro
```

Paste this:

```xml
<?xml version="1.0"?>
<robot name="1dof_arm" xmlns:xacro="http://www.ros.org/wiki/xacro">

  <!-- Basic dimensions -->
  <xacro:property name="base_size" value="0.2"/>
  <xacro:property name="link_length" value="0.5"/>
  <xacro:property name="link_radius" value="0.04"/>

  <!-- Base link -->
  <link name="base_link">
    <visual>
      <origin xyz="0 0 0.05" rpy="0 0 0"/>
      <geometry>
        <box size="${base_size} ${base_size} 0.1"/>
      </geometry>
      <material name="blue">
        <color rgba="0.1 0.3 0.8 1.0"/>
      </material>
    </visual>
  </link>

  <!-- Arm link -->
  <link name="link1">
    <visual>
      <origin xyz="0 0 ${link_length/2.0}" rpy="0 0 0"/>
      <geometry>
        <cylinder radius="${link_radius}" length="${link_length}"/>
      </geometry>
      <material name="orange">
        <color rgba="1.0 0.5 0.1 1.0"/>
      </material>
    </visual>
  </link>

  <!-- Revolute joint -->
  <joint name="joint1" type="revolute">
    <parent link="base_link"/>
    <child link="link1"/>
    <origin xyz="0 0 0.1" rpy="0 0 0"/>
    <axis xyz="0 1 0"/>
    <limit lower="-1.57" upper="1.57" effort="10" velocity="1.0"/>
  </joint>

</robot>
```

URDF describes the robot’s links, joints, geometry, and organization, while Xacro is the recommended way to reduce repetition and make URDF files easier to maintain.

---

# 3. Why this robot works

This is the meaning of the model:

* `base_link` is the fixed base.
* `link1` is the arm segment.
* `joint1` connects them.
* `axis xyz="0 1 0"` means the joint rotates about the **y-axis**.
* the cylinder is drawn with its center in the middle, so we shifted its visual origin upward by `link_length/2`.

That last point is very important: in URDF, the geometry is often centered at its own local frame, so you usually offset the visual origin to place the link where you expect it. The URDF tutorials emphasize getting the geometry and frame relationships correct first before adding more features. ([docs.ros.org][1])

Open:

```bash
cd ~/mybot_ws/src/my_arm_description/rviz

```

Paste this minimal config:

```yaml
Panels:
  - Class: rviz_common/Displays
    Name: Displays
  - Class: rviz_common/Selection
    Name: Selection
  - Class: rviz_common/Tool Properties
    Name: Tool Properties
  - Class: rviz_common/Views
    Name: Views
Visualization Manager:
  Class: ""
  Displays:
    - Class: rviz_default_plugins/Grid
      Name: Grid
      Enabled: true
    - Class: rviz_default_plugins/RobotModel
      Name: RobotModel
      Enabled: true
      Description Source: Topic
      Description Topic:
        Value: /robot_description
    - Class: rviz_default_plugins/TF
      Name: TF
      Enabled: true
  Global Options:
    Fixed Frame: base_link
  Name: root
  Tools:
    - Class: rviz_default_plugins/Interact
    - Class: rviz_default_plugins/MoveCamera
    - Class: rviz_default_plugins/Select
  Views:
    Current:
      Class: rviz_default_plugins/Orbit
```

Set the fixed frame to `base_link`. That avoids one of the most common RViz problems for a new robot: the robot exists, but RViz is using the wrong fixed frame. RViz visualizes the robot from the TF tree being published by `robot_state_publisher`. ([docs.ros.org][2])

# 5. Create the launch file

Open:

```bash
cd ~/mybot_ws/src/my_arm_bringup/launch
gedit display.launch.py
```

Paste this:

```python
from launch import LaunchDescription
from launch.substitutions import Command, PathJoinSubstitution
from launch_ros.actions import Node
from launch_ros.parameter_descriptions import ParameterValue
from launch_ros.substitutions import FindPackageShare
from launch.substitutions import FindExecutable


def generate_launch_description():
    xacro_file = PathJoinSubstitution([
        FindPackageShare("my_arm_description"),
        "urdf",
        "1dof_arm.urdf.xacro"
    ])

    robot_description = ParameterValue(
        Command([FindExecutable(name="xacro"), " ", xacro_file]),
        value_type=str
    )

    rviz_config = PathJoinSubstitution([
        FindPackageShare("my_arm_description"),
        "rviz",
        "view_arm.rviz"
    ])

    return LaunchDescription([
        Node(
            package="joint_state_publisher_gui",
            executable="joint_state_publisher_gui",
            name="joint_state_publisher_gui"
        ),
        Node(
            package="robot_state_publisher",
            executable="robot_state_publisher",
            name="robot_state_publisher",
            parameters=[{"robot_description": robot_description}]
        ),
        Node(
            package="rviz2",
            executable="rviz2",
            name="rviz2",
            arguments=["-d", rviz_config]
        )
    ])
```

ROS 2 launch files are meant to start and configure multiple nodes together, and Python launch files are a standard way to do that in Humble. Using Xacro in the launch file to generate the `robot_description` parameter is the normal pattern for URDF/Xacro-based robot bringup. ([docs.ros.org][5])

# 6. Make sure the files get installed

You need both packages to install their resources correctly.

## 6.1 Update `my_arm_description/CMakeLists.txt`

Open:

```bash
cd ~/mybot_ws/src/my_arm_description
gedit CMakeLists.txt
```

Add this near the end, before `ament_package()`:

```cmake
install(
  DIRECTORY urdf rviz meshes
  DESTINATION share/${PROJECT_NAME}
)
```

A description package must install its URDF/Xacro and RViz resources into the package share directory so that launch files can find them through package lookup. That matches the standard ROS 2 package-and-launch workflow. ([docs.ros.org][3])

## 6.2 Update `my_arm_bringup/setup.py`

Open:

```bash
cd ~/mybot_ws/src/my_arm_bringup
gedit setup.py
```

Find `data_files=[ ... ]` and make sure it looks like this:

```python
from setuptools import setup
from glob import glob
import os

package_name = 'my_arm_bringup'

setup(
    name=package_name,
    version='0.0.0',
    packages=[package_name],
    data_files=[
        ('share/ament_index/resource_index/packages',
            ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
        (os.path.join('share', package_name, 'launch'), glob('launch/*.py')),
    ],
    install_requires=['setuptools'],
    zip_safe=True,
    maintainer='YOUR_NAME',
    maintainer_email='your@email.com',
    description='Bringup package for my arm',
    license='TODO: License declaration',
    tests_require=['pytest'],
    entry_points={
        'console_scripts': [],
    },
)
```

For Python packages, launch files are typically stored in `launch/` and installed into the package’s share directory so `ros2 launch` can find them after building the workspace. ([docs.ros.org][3])

# 9. Common problems and fixes

## Problem A: robot does not appear

Check:

```bash
ros2 param get /robot_state_publisher robot_description
```

If the parameter exists, then the robot model is loaded. `robot_state_publisher` depends on a valid robot model and incoming joint states to publish the TF tree correctly. ([docs.ros.org][2])

## Problem B: RViz says fixed frame error

Set:

```text
Fixed Frame = base_link
```

The fixed frame must be a valid frame from the robot’s TF tree. ([docs.ros.org][2])

## Problem C: package not found

Rebuild and source again:

```bash
cd ~/mybot_ws
source /opt/ros/humble/setup.bash
colcon build
source install/setup.bash
```

ROS 2 workspaces use the underlay/overlay model, so forgetting to source the workspace is a common reason packages or launch files are not found. ([docs.ros.org][6])

## Problem D: xacro executable not found

Install it:

```bash
sudo apt install -y ros-humble-xacro
```

Xacro is the standard tool used to generate URDF from `.xacro` files. ([docs.ros.org][7])
