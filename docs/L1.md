# 2. The package structure you should use

For manipulation projects, do **not** put everything into one package.

Use this style:

```text
~/mybot_ws/src/
├── my_robot0_description/
│   ├── urdf/
│   │   └── 1dof_arm.urdf.xacro
|   |   └── 2dof_arm.urdf.xacro
│   └── rviz/
│       └── view_1dof_arm.rviz
└── my_robot0_bringup/
    └── launch/
        └── display_1dof_arm.launch.py
        └── display_2dof_arm.launch.py
```

What each one does:

* `my_robot0_description`

  * URDF/Xacro
  * meshes
  * RViz config
* `my_robot0_bringup`

  * launch files
  * robot_state_publisher
  * RViz launch
  * Gazebo launch
* `my_robot0_control`

  * controller config YAML
  * control-related launch or helper nodes
* `my_robot0_kinematics`

  * FK and IK Python scripts or nodes

For the first few lessons, we only create:

* `my_robot0_description`
* `my_robot0_bringup`

That is enough to begin correctly.

# 3. Install the basic packages

Run this first:

```bash
sudo apt update
sudo apt install -y \
  ros-humble-xacro \
  ros-humble-joint-state-publisher-gui \
  ros-humble-robot-state-publisher \
  ros-humble-rviz2 \
  ros-humble-ros2-control \
  ros-humble-ros2-controllers
```

These packages match the Humble docs for Xacro and `ros2_control`. 

If later you want Gazebo Sim integration with `ros2_control`, the Humble docs also provide `gz_ros2_control` for connecting Gazebo Sim and the controller manager. 

---

# 4. Create your workspace

Run:

```bash
mkdir -p ~/mybot_ws/src
cd ~/mybot_ws
source /opt/ros/humble/setup.bash
colcon build
source install/setup.bash
```

Why this matters:

* `/opt/ros/humble` is your **underlay**
* `~/mybot_ws/install` becomes your **overlay**
* your own packages override or extend the base ROS install

That overlay workflow is the recommended ROS 2 approach. 

---

# 5. Create the first two packages

Go into `src`:

```bash
cd ~/mybot_ws/src
```

Create the description package:

```bash
ros2 pkg create my_robot0_description
```

Create the bringup package:

```bash
ros2 pkg create my_robot0_bringup --build-type ament_cmake
```

ROS 2 Humble docs show package creation from inside the workspace `src` directory, and launch files are commonly integrated into packages as part of the normal workflow. 

---

# 6. Make the folders you will need

Run:

```bash
cd ~/mybot_ws/src/my_robot0_description
mkdir -p urdf meshes rviz

cd ~/mybot_ws/src/my_robot0_bringup
mkdir -p launch config
```

Now your structure is:

```text
~/mybot_ws/src/
├── my_robot0_description/
│   ├── urdf/
│   ├── meshes/
│   └── rviz/
└── my_robot0_bringup/
    ├── launch/
    └── config/
```

This is the cleanest start for robot projects.

# 7. Build once to confirm the workspace is healthy

```bash
cd ~/mybot_ws
source /opt/ros/humble/setup.bash
colcon build
source install/setup.bash
```

Then check that ROS 2 can see your packages:

```bash
ros2 pkg list | grep my_robot0
```

You should see:

```text
my_robot0_bringup
my_robot0_description
```

If you see them, your foundation is correct.

# 8. The mental model you must remember

This is the most important concept today:

## `my_robot0_description`

This package answers:

> What is the robot?

It contains the robot model:

* links
* joints
* geometry
* inertial
* collision
* robot structure

## `my_robot0_bringup`

This package answers:

> How do I run the robot?

It contains:

* launch files
* RViz startup
* robot_state_publisher
* later Gazebo
* later controllers

This separation will save you a lot of confusion later.

# 11. What to do if there is an error

## If `ros2: command not found`

Run:

```bash
source /opt/ros/humble/setup.bash
```

## If `colcon` fails

Check:

```bash
echo $ROS_DISTRO
which colcon
```

## If package is created but not found after build

Run again:

```bash
cd ~/mybot_ws
source /opt/ros/humble/setup.bash
colcon build
source install/setup.bash
```
