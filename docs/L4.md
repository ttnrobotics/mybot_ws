The reason for this lesson is simple: once a URDF starts growing, it becomes hard to read and easy to break. The ROS 2 Humble Xacro tutorial introduces Xacro specifically to reduce repetition and make robot descriptions easier to read and maintain, and the launch/large-project docs emphasize structuring files for reuse as projects grow.

# Lesson 4 — Clean up the robot using Xacro properly

## Goal of this lesson

By the end of this lesson, you will:

* keep the **same 2-DOF robot**
* split the robot description into cleaner files
* use **Xacro properties**
* use **Xacro macros**
* understand why this structure is better for larger robots
* keep your existing RViz launch working

This lesson is not about adding new robot capability.
It is about making the robot description **organized, reusable, and scalable**.

---

# 1. Why we need this lesson

Right now, your robot file probably works, but everything is in one file:

* dimensions
* masses
* colors
* link definitions
* joint definitions

That is okay for a tiny robot.
It becomes painful for:

* 6-DOF or 7-DOF arms
* robots with repeated link patterns
* robots with different variants
* later `ros2_control` and Gazebo tags

The ROS 2 Humble Xacro tutorial exists exactly for this reason: use properties, macros, and cleaner structure so the description is easier to maintain. ([ROS][1])

---

# 2. What you will build in this lesson

We will change from this style:

```text
my_first_arm.urdf.xacro
  └── everything inside one file
```

to this style:

```text
urdf/
├── my_first_arm.urdf.xacro
├── arm_properties.xacro
└── arm_macros.xacro
```

## Meaning of each file

### `arm_properties.xacro`

Stores shared values:

* link lengths
* base size
* radius
* masses
* colors if needed

### `arm_macros.xacro`

Stores reusable robot building blocks:

* macro for a cylindrical link
* macro for a revolute joint

### `my_first_arm.urdf.xacro`

Main robot assembly file:

* includes other Xacro files
* creates the robot by calling macros
* stays short and readable

This is the same general direction recommended by the Xacro and large-project launch/tutorial structure: make files reusable and separate responsibilities. ([ROS][1])

---

# 3. The key Xacro ideas you need to understand first

Before touching files, understand these three ideas.

## A. Property

A property is like a named variable.

Example:

```xml
<xacro:property name="link1_length" value="0.45"/>
```

Then later:

```xml
${link1_length}
```

This avoids repeating numbers in many places. The ROS 2 Xacro tutorial explicitly uses properties to simplify and clean robot description files. ([ROS][1])

---

## B. Macro

A macro is like a reusable function for XML.

Example idea:

```xml
<xacro:my_link name="link1" length="0.45" radius="0.04" mass="0.5"/>
```

Instead of writing `visual`, `collision`, and `inertial` blocks again and again, you write them once inside a macro and reuse them. Xacro’s purpose is exactly this kind of reuse and reduction of repetitive URDF code. ([ROS][1])

---

## C. Include

If you split the robot into multiple Xacro files, the main file can include the others.

That keeps the project cleaner and matches the “organize for reuse” idea in ROS 2 documentation. ([ROS][1])

---

# 4. Final file structure after this lesson

Inside `my_arm_description/urdf`, you will have:

```text
urdf/
├── arm_properties.xacro
├── arm_macros.xacro
└── my_first_arm.urdf.xacro
```

# 5. Create the properties file

Open:

```bash
cd ~/arm_ws/src/my_arm_description/urdf
gedit arm_properties.xacro
```

Paste this:

```xml
<?xml version="1.0"?>
<robot xmlns:xacro="http://www.ros.org/wiki/xacro">

  <!-- Base dimensions -->
  <xacro:property name="base_x" value="0.20"/>
  <xacro:property name="base_y" value="0.20"/>
  <xacro:property name="base_z" value="0.10"/>

  <!-- Link dimensions -->
  <xacro:property name="link1_length" value="0.45"/>
  <xacro:property name="link2_length" value="0.35"/>
  <xacro:property name="link_radius" value="0.04"/>

  <!-- Masses -->
  <xacro:property name="base_mass" value="2.0"/>
  <xacro:property name="link1_mass" value="0.5"/>
  <xacro:property name="link2_mass" value="0.4"/>

  <!-- Colors -->
  <xacro:property name="base_color" value="0.1 0.3 0.8 1.0"/>
  <xacro:property name="link1_color" value="1.0 0.5 0.1 1.0"/>
  <xacro:property name="link2_color" value="0.2 0.8 0.3 1.0"/>

</robot>
```

## Why this file matters

This file centralizes the robot’s dimensions and masses.
So later, if you want to change the arm length or mass, you do it in one place instead of searching the whole file.

That is one of the main benefits of Xacro described in the ROS 2 tutorial. ([ROS][1])

# 6. Create the macros file

Open:

```bash
gedit arm_macros.xacro
```

Paste this:

```xml
<?xml version="1.0"?>
<robot xmlns:xacro="http://www.ros.org/wiki/xacro">

  <!-- Reusable cylindrical arm link -->
  <xacro:macro name="arm_link" params="name length radius mass color_rgba">

    <link name="${name}">

      <visual>
        <origin xyz="0 0 ${length/2.0}" rpy="0 0 0"/>
        <geometry>
          <cylinder radius="${radius}" length="${length}"/>
        </geometry>
        <material name="${name}_material">
          <color rgba="${color_rgba}"/>
        </material>
      </visual>

      <collision>
        <origin xyz="0 0 ${length/2.0}" rpy="0 0 0"/>
        <geometry>
          <cylinder radius="${radius}" length="${length}"/>
        </geometry>
      </collision>

      <inertial>
        <origin xyz="0 0 ${length/2.0}" rpy="0 0 0"/>
        <mass value="${mass}"/>
        <inertia
          ixx="0.01" ixy="0.0" ixz="0.0"
          iyy="0.01" iyz="0.0"
          izz="0.0004"/>
      </inertial>

    </link>

  </xacro:macro>

  <!-- Reusable revolute joint -->
  <xacro:macro name="revolute_joint" params="name parent child xyz rpy axis lower upper effort velocity damping friction">

    <joint name="${name}" type="revolute">
      <parent link="${parent}"/>
      <child link="${child}"/>
      <origin xyz="${xyz}" rpy="${rpy}"/>
      <axis xyz="${axis}"/>
      <limit lower="${lower}" upper="${upper}" effort="${effort}" velocity="${velocity}"/>
      <dynamics damping="${damping}" friction="${friction}"/>
    </joint>

  </xacro:macro>

</robot>
```

---

# 7. Understand the two macros carefully

## Macro 1: `arm_link`

This macro creates one cylindrical link with:

* visual
* collision
* inertial

So instead of rewriting the same block for `link1`, `link2`, `link3`, and so on, you reuse the same template.

That is exactly the kind of repetition Xacro is designed to remove. ([ROS][1])

## Macro 2: `revolute_joint`

This macro creates a revolute joint with:

* parent
* child
* origin
* axis
* limits
* dynamics

So later, making more joints becomes much faster and less error-prone.

# 8. Replace the main robot file

Now open:

```bash
gedit my_first_arm.urdf.xacro
```

Replace everything with this:

```xml
<?xml version="1.0"?>
<robot name="my_first_arm" xmlns:xacro="http://www.ros.org/wiki/xacro">

  <!-- Include helper files -->
  <xacro:include filename="$(find my_arm_description)/urdf/arm_properties.xacro"/>
  <xacro:include filename="$(find my_arm_description)/urdf/arm_macros.xacro"/>

  <!-- Base link -->
  <link name="base_link">

    <visual>
      <origin xyz="0 0 ${base_z/2.0}" rpy="0 0 0"/>
      <geometry>
        <box size="${base_x} ${base_y} ${base_z}"/>
      </geometry>
      <material name="base_material">
        <color rgba="${base_color}"/>
      </material>
    </visual>

    <collision>
      <origin xyz="0 0 ${base_z/2.0}" rpy="0 0 0"/>
      <geometry>
        <box size="${base_x} ${base_y} ${base_z}"/>
      </geometry>
    </collision>

    <inertial>
      <origin xyz="0 0 ${base_z/2.0}" rpy="0 0 0"/>
      <mass value="${base_mass}"/>
      <inertia
        ixx="0.008333" ixy="0.0" ixz="0.0"
        iyy="0.008333" iyz="0.0"
        izz="0.013333"/>
    </inertial>

  </link>

  <!-- Link 1 -->
  <xacro:arm_link
    name="link1"
    length="${link1_length}"
    radius="${link_radius}"
    mass="${link1_mass}"
    color_rgba="${link1_color}"/>

  <!-- Link 2 -->
  <xacro:arm_link
    name="link2"
    length="${link2_length}"
    radius="${link_radius}"
    mass="${link2_mass}"
    color_rgba="${link2_color}"/>

  <!-- Joint 1 -->
  <xacro:revolute_joint
    name="joint1"
    parent="base_link"
    child="link1"
    xyz="0 0 ${base_z}"
    rpy="0 0 0"
    axis="0 1 0"
    lower="-1.57"
    upper="1.57"
    effort="20"
    velocity="1.0"
    damping="0.2"
    friction="0.1"/>

  <!-- Joint 2 -->
  <xacro:revolute_joint
    name="joint2"
    parent="link1"
    child="link2"
    xyz="0 0 ${link1_length}"
    rpy="0 0 0"
    axis="0 1 0"
    lower="-1.57"
    upper="1.57"
    effort="20"
    velocity="1.0"
    damping="0.2"
    friction="0.1"/>

</robot>
```

# 9. What changed, conceptually

Before, the main file contained everything.

Now the main file does mainly three things:

* includes helper files
* defines the unique base link
* assembles the robot using macros

This is much closer to how larger robot description packages are organized in practice, and it aligns with ROS 2’s guidance to structure things for readability and reuse. ([ROS][1])

# 10. Why the base link is not a macro yet

You may ask: why not make the base a macro too?

You can.
But for teaching, it is better to keep one part explicit.

The base link is a good place to still see a normal full URDF block, while the repeating arm links are moved into macros.

That way you understand both:

* plain URDF structure
* reusable Xacro structure

# 14. Common mistakes in this lesson

## Mistake 1: wrong include path

If the include path is wrong, Xacro will fail before RViz starts.

Check this carefully:

```xml
<xacro:include filename="$(find my_arm_description)/urdf/arm_properties.xacro"/>
<xacro:include filename="$(find my_arm_description)/urdf/arm_macros.xacro"/>
```

## Mistake 2: macro name typo

If you define:

```xml
<xacro:macro name="arm_link" ...>
```

then you must call:

```xml
<xacro:arm_link ... />
```

not another name.

## Mistake 3: property name typo

For example:

* `link1_length` vs `link_1_length`
* `link_radius` vs `radius`

Xacro is very sensitive to exact names.

## Mistake 4: missing self-closing slash

When calling a macro, use:

```xml
<xacro:arm_link ... />
```

not an unfinished tag.

# 15. A very useful debug step

If launch fails, test the Xacro directly.

Run:

```bash
ros2 run xacro xacro ~/arm_ws/src/my_arm_description/urdf/my_first_arm.urdf.xacro
```

If Xacro has a syntax or include problem, this command usually shows the error more directly.

The Humble documentation treats Xacro as the normal preprocessing step before URDF is used by the rest of the system. ([ROS][1])
